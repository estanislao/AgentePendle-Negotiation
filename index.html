Módulo de Negociações Meta-Orientado (PT Only)

1. Visão Geral da Arquitetura

A arquitetura da nova aba "Negociações" foi projetada como um motor analítico e transacional reativo acoplado ao ecossistema Pendle na Arbitrum. O sistema opera em um fluxo unidirecional composto por três camadas lógicas:

  • Camada de Ingestão de Dados (Data Ingestion): Consome em tempo real via RPC MEV-Shielded e Subgraph os saldos da carteira do usuário (AUM) e os dados de mercado das pools Pendle (Fixed APYs e Maturidades).
  • Motor de Decisão e Otimização (Core Engine): Centraliza as regras matemáticas meta-orientadas e os limites de segurança estruturais (guardrails de 50% para Swap e 20% para Buy).
  • Camada de Simulação e Projeção (UI/UX State): Renderiza o impacto futuro da carteira antes da assinatura on-chain, evidenciando o deslocamento do portfólio em direção ao alvo estipulado.

2. Estrutura de Dados e Motores Cripto (Backend)

O backend utiliza um motor analítico focado em **Média Ponderada de Rendimento** acoplado a regras rígidas de barreira (Circuit Breakers/Guardrails). O objetivo principal é manter o portfólio direcionado à meta de 13.00% aa.

2.1. Regras de Negócio e Controle de Risco (Guardrails)

  • Swap Gate: Limita o valor máximo da sugestão a 50% do capital atualmente imobilizado em outras posições de PT. O motor varre e seleciona para descarte preferencial as pools com menor rendimento ou cujas curvas de decaimento de liquidez (PDS) degradaram.
  • Buy Gate: Limita o valor sugerido para compras via capital livre (caixa/stablecoins) a no máximo 20% do AUM total do portfólio.

2.2. Formulação Matemática do Motor

O rendimento consolidado da carteira é recalculado dinamicamente a cada simulação através da fórmula:

APY_{Portfolio} = \sum_{i=1}^{n} \left( \frac{Valor\_Alocado_i}{AUM\_Total} \times Fixed\_APY_i \right)

O Ganho Real Líquido ($\Delta_{Net}$) de uma troca de ativos calcula o diferencial de rendimento projetado até a data de vencimento, subtraindo os custos friccionais de rede:

Ganho\_Real = (Valor\_Swap \times APY_{Nova} \times t) - (Valor\_Swap \times APY_{Atual} \times t) - C_{Saída}

Onde t = Dias_Restantes_Maturidade / 365 e C_Saída representa o somatório do Gas da rede mais o slippage gerado na desmontagem.

2.3. Lógica de Roteamento Multihop (Pendle Router)

Como as pools do Pendle não possuem swaps diretos do tipo PT_A ➔ PT_B, o motor cripto monta e codifica em lote (Multicall) a seguinte rota atômica de contratos inteligentes:

[PT_Origem] ➔ swapExactPtForToken() ➔ [Underlying Asset / SY] ➔ swapExactTokenForPt() ➔ [PT_Destino]

3. Especificação dos Endpoints & Integrações (Pendle)

O Agente precisará consumir as seguintes interfaces de dados estruturados para alimentar o motor analítico:

3.1. Subgraph / API GraphQL da Pendle (Dados de Mercado)

Para buscar as condições em tempo real dos mercados elegíveis da rede Arbitrum:

query GetPendleMarkets {
  markets(where: { chainId: 42161, isExpired: false }) {
    id
    ptAddress
    expiry
    ptPricing {
      fixedApy
      impliedApy
    }
    underlyingAsset {
      id
      symbol
    }
  }
}

3.2. Contrato Inteligente On-Chain (Pendle Router V4)

Para estimar o custo exato de saída de uma PT e entrada na nova (simulação de Slippage), o backend deve fazer chamadas estáticas (eth_call) nos métodos de leitura do Router oficial (Bypass de transação):

  • IAllActionV3.simRedeemPyToToken.call(...): Retorna o volume exato recebido do ativo base ao queimar a PT antiga.
  • IAllActionV3.simMintPyFromToken.call(...): Retorna a quantidade exata de novas PTs obtidas na pool recomendada com base nos tokens da primeira etapa.
